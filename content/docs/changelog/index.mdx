Backward-compatible enhancements only — no breaking changes since the initial
release.

## 1.6.1 — Error code documentation

Documented the `errors` array responses across Payout, OTC Quote, and OTC
Place Order with the machine-readable codes the API has been returning:

- `account.payout.duplicate_reference_id_#<id>` — duplicate
  `client_reference_id` on Create Payout.
- `market.otc_quote.duplicate_reference_id_#<id>` — duplicate
  `client_reference_id` on Create OTC Quote.
- `market.otc_quote.not_found` — `quote_id` is unknown (Place OTC Order +
  propagated through Create Payout when `otc_quote_id` is set).
- `market.otc_quote.amount_mismatch` — quote amount no longer matches
  (Create Payout only, when the quote drifts between create and execution).
- `market.otc_quote.expired` — quote is past expiry (Place OTC Order +
  propagated through Create Payout when `otc_quote_id` is set).

Payout `400` / `401` responses now use the standard `errors: [...]` array
shape rather than the older singular `error: "..."` field.

## 1.6.0 — `client_reference_id` on OTC quote & conversion order

OTC quote and the "place order from quote" endpoint now accept an optional
`client_reference_id`, so you can correlate your own ledger entries with both
the quote and the order it converts into.

- `POST /market/otc_quotes/new` — request gains optional `client_reference_id`
  (max 64 chars). Echoed back on the quote.
- `OtcQuoteResponse` — gains `client_reference_id` field (nullable).
- `PATCH /market/otc_quotes/place_order/{quote_id}` — request body is now
  accepted (still optional) and can carry `client_reference_id` for the order
  created from the quote. If omitted, the order inherits the quote's reference
  (when set), otherwise the platform generates one.

Existing integrations keep working unchanged — both fields are optional.

## 1.5.0 — OTC-funded payouts

Create Payout (`POST /account/payouts/initiate`) now accepts an optional OTC
funding leg. When `otc_quote_id` is provided, the API places an order against
that previously-created OTC quote to fund the payout in a single request; the
executed order is returned alongside the payout under `otc_order`.

### Request — new optional field
- `otc_quote_id` — ID of an active OTC quote (created via
  `POST /market/otc_quotes/new`) to fund the payout from. Omit to settle from
  existing balance.

### Response — additions
- `otc_order` — embedded `OtcOrderResponse`, present only when the request
  included `otc_quote_id`. Its `client_reference_id` is automatically
  populated with the generated `payout_id` so the two records correlate
  cleanly.
- `remark` — free-form note attached to the payout, typically mirroring
  `status_code` for in-flight payouts.

## 1.4.1 — Documentation corrections

No API behaviour changed in this release — the backend already worked this way.
These entries bring the spec in line with what the API has been returning all
along.

### OTC quote & order
- `req_currency` is now documented on `POST /market/otc_quotes/new` and
  `POST /market/otc_orders` (request) and on the quote response. It names the
  currency that `req_amount` / `amount` is denominated in (typically the fiat
  side of the market, e.g. `inr` for `usdtinr`).

### Payouts
- `completed_at` documented as **integer** (milliseconds since epoch), matching
  `created_at`'s shape. The spec previously declared it as an ISO 8601 string;
  the API has always returned a millisecond integer.

### Beneficiaries
- Added `FAILED` to the `verification_data_bank_transaction_status` enum (now
  one of `PROCESSING`, `VERIFIED`, `NOT_FOUND`, `FAILED`).

## 1.4.0 — UPI payment support for beneficiaries & payouts

### Beneficiary management
Added support for **UPI beneficiaries** to both Create and Get endpoints.
Existing bank-account integrations require no changes.

- `vpa` — create or filter by UPI ID (for example, `user@okaxis`).
- `vpa` is mutually exclusive with `account_number` and `bank_ifsc_code`.

### Payout initiation
Added support for **UPI payouts** to the Create Payout endpoint. Existing
bank-payout integrations require no changes.

- `vpa` — initiate a payout to a UPI ID.
- `vpa` is mutually exclusive with `account_number` and `bank_ifsc_code`.
- `account_number` and `bank_ifsc_code` must be provided together or not at all.
- At least one of `vpa` or `account_number` must be provided.

## 1.3.0 — Enhancements & new filters

### Payout filters
New query parameters for better transaction traceability:

- `bank_transaction_id` — filter payouts using the bank-assigned transaction ID.
- `bank_currency` — filter payouts by fiat currency.

### Payout creation
Optional bank-account verification:

- `penny_drop_required` (boolean, default `true`) — whether a penny-drop
  verification runs before processing the payout.

### OTC quote API
- `quote_id` — retrieve or filter using the unique quote identifier.

## 1.2.0 — Flexible date-range filtering

You can now filter list endpoints by date range using:

- `range` — selects the date field (`created` or `updated`, default `created`).
- `from` — start datetime (ISO 8601).
- `to` — end datetime (ISO 8601).

Example:

```
/payouts?range=created&from=2025-09-30T00:00:00Z&to=2025-09-30T23:59:59Z
```

If only `from` is provided, results from that time onward are returned. If only
`to` is provided, results up to that time are returned.

## 1.1.1 — Beneficiary IFSC filter support

*Released 23 Feb 2026.*

The Beneficiary GET endpoint now accepts `bank_ifsc_code`.

- With `account_number`: returns the exact match for `account_number + bank_ifsc_code`.
- Without `account_number`: returns all records matching the IFSC code.

Backward-compatible — does not modify existing response structures.

## 1.1.0 — Historical order enhancements

*Released 20 Feb 2026.*

GET endpoints (Payout, Beneficiary, OTC) now support:

- Pagination — `page`, `limit`.
- Sorting — `order_by` (`id`, `created_at`, `updated_at`), `ordering` (`asc`, `desc`, default `desc`).

Pagination metadata is returned via response headers: `Total`, `Page`, `Per-Page`.
